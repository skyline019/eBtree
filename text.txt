第二部分：磁盘核心数据布局（Page Layout）

1. Super Block (每个分片 2个槽位, 各4KB)

```cpp
struct SuperBlock {
    uint64 magic;           // 0xEBE5BEEF
    uint64 epoch;           // 全局单调递增版本号 (选最大且CRC校验通过者)
    uint64 root_offset;     // B-Tree 根节点物理偏移
    uint64 tlog_tail;       // 时间链尾部偏移 (用于快速闪回)
    uint64 data_lsn;        // 最后落盘数据页的最大LSN
    uint8  reserved[4056];  // 预留扩展
    uint64 crc64;           // 整块校验 (保障扇区原子性)
};
```

2. B-Tree 内部节点 (页头64字节, 固定)

```cpp
struct InternalHeader {
    uint8  type;            // 0x01 = Internal
    uint16 key_count;       // 子节点指针数量 (通常 100~200)
    uint8  summary_type;    // 0=MinMax, 1=直方图, 2=前缀树
    uint64 summary_lsn;     // 摘要版本 (用于检测漂移, 触发自愈)
    // 自适应摘要区 (64B - 16B = 48B 可变)
    union {
        struct { uint64 min; uint64 max; } mm;          // 整数
        struct { uint32 prefix_hash; uint32 suffix_min; } trie; // 字符串
    } summary;
    uint32 child_offsets[1]; // 紧随其后的子节点偏移数组 (变长)
};
```

3. 时间链 (T-Log) 记录 (仅追加, 每记录16字节)

```cpp
struct TLogEntry {
    uint64 page_offset;     // 指向数据页 (0表示墓碑/删除)
    uint32 timestamp_sec;   // 写入时间戳 (用于闪回窗口)
    uint32 prev_offset;     // 逻辑前驱在T-Log文件中的偏移 (构成链表)
};
```

第三部分：核心读写路径流程图

写入路径 (极速流水线)

```text
App Write -> Rust 路由 (Hash取模) 
          -> 定位分片 MemTable (无锁哈希插入) 
          -> 异步 Group Commit 刷 WAL (合并fsync) 
          -> 后台 Flusher 顺序追加至 .data 文件尾 
          -> 追加 T-Log 记录 (原子) 
          -> 原子更新 SuperBlock.epoch + root_offset (双槽轮换)
```

读取路径 (No Fallback 铁律)

```text
App Query (含 WHERE) -> Rust Prepare 校验 (检查CUE元数据，列是否可查)
                     -> 并行路由到 1~N 个分片 
                     -> C++ 内核从根节点遍历:
                         1. 读内部节点摘要 (mmap零拷贝)
                         2. 若摘要不满足条件 -> 直接裁剪返回 (No Fallback)
                         3. 若摘要过时 (LSN漂移) -> 触发懒惰自愈 (安全降级)
                         4. 下钻至叶子 -> 批量返回偏移数组
                     -> Rust 并行 mmap 读取具体数据页 (RCU窗口)
                     -> 压缩并返回给 App
```

第四部分：恢复与一致性状态机（高可用保障）

```text
[系统崩溃/重启] -> 加载每个分片双槽 SuperBlock 
                  -> 选择最大 Epoch 且 CRC64 通过者 (原子恢复)
                  -> 检查 root_offset 指向的 B-Tree 根节点 CRC
                     ├── 成功 -> 立即提供服务 (<15ms)
                     └── 失败 -> 触发 "懒惰恢复"：
                          1. 不重建全量索引
                          2. 查询时按需从 WAL 中截取物理镜像恢复损坏页
                          3. 若 WAL 损坏，从 T-Log 定位上一个快照点
                  -> 后台异步：校验所有内部节点摘要，修复漂移
                  -> 状态切换至 "Hot" 就绪
```

第五部分：完整开发规划（3个月 MVP）

阶段 周期 核心交付物 验收标准
Phase 1：地基 Week 1-2 C++ 分片管理、双槽 SuperBlock 原子 IO、RCU mmap 窗口 千万级数据写入不崩溃，断电恢复数据零丢失 (通过 fsync 测试)
Phase 2：核心读写 Week 3-4 B-Tree 遍历 (模板化)、自适应摘要 (MinMax/Trie)、SIMD 裁剪 点查 < 1ms，范围扫描 (带过滤) 不扫描无关页 (验证 No Fallback)
Phase 3：流水线与时间链 Week 5-6 3段 MemTable 流水线、T-Log 时间链实现、物理闪回查询 写入吞吐 > 50k TPS，历史查询误差 < 1s
Phase 4：恢复与自愈 Week 7-8 懒惰恢复引擎、摘要自愈触发器、GC 冷热分级迁移 崩溃恢复 < 80ms (最坏坏块场景)，摘要漂移自动修复
Phase 5：Rust/CUE 封装 Week 9-10 Rust FFI 批量接口、CUE 编译时生成、SQL 准备器校验 跨语言调用开销 < 5%，CUE 配置变更无需重启
Phase 6：高压混沌测试 Week 11-12 模拟千万级数据 + 1000并发 + 随机掉电 + 磁盘坏块注入 性能抖动 < 10%，零数据逻辑错误

第六部分：最终性能承诺书（生产环境基线）

· 资源占用：RAM < 32MB (含双缓冲窗口)，ROM < 2MB (代码段)。
· 写入：稳定 60,000 TPS (无Compaction停顿，P99 < 300μs)。
· 点查：延迟 0.7ms (P99)，命中即返回，绝不回退。
· 范围扫描(1000行)：5.5ms (并行IO + T-Log辅助)。
· 恢复时间目标 (RTO)：< 80ms (通过双槽原子恢复 + 懒惰重放)。
· 恢复点目标 (RPO)：0 (物理全量日志 + 双WAL轮换，保证数据零丢失)。
